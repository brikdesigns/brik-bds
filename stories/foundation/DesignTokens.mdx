# Design Tokens

Design tokens are the single source of truth for every visual decision in the Brik ecosystem — color, type, spacing, radius, shadow, motion. A token gives a value a name and a purpose. Components never reference raw values; they reference tokens. This means a component built once works correctly across every Brik client without touching its code.

---

## Three-tier library hierarchy

Every project in the Brik ecosystem sits in a three-tier stack:

```
┌─────────────────────────────────────────────────────────────────────────┐
│  TIER 1 — BDS Foundations  (this repo)                                  │
│  brik-bds · github.com/brikdesigns/brik-bds                             │
│                                                                         │
│  Primitive scale tokens — the raw numbers and colors that never change  │
│  between clients. Consumed by Tier 2 and Tier 3 via CSS cascade.        │
│                                                                         │
│  --color-poppy-light  --space-400  --font-size-700  --border-radius-200  │
└──────────────────────────────┬──────────────────────────────────────────┘
                               │  CSS cascade (submodule import)
┌──────────────────────────────▼──────────────────────────────────────────┐
│  TIER 2 — Brik Brand Kit  (overrides.css + theme blocks)                │
│                                                                         │
│  Semantic tokens — purpose-bound aliases that map to Tier 1 primitives. │
│  These are what components use. Values change per theme; names never do. │
│                                                                         │
│  --background-brand-primary  --text-primary  --padding-lg  --shadow-sm  │
│                                                                         │
│  9 built-in themes ship here. Each .body.theme-N block overrides the    │
│  semantic color tokens for that brand without touching any component.   │
└──────────────────────────────┬──────────────────────────────────────────┘
                               │  CSS cascade (client globals.css)
┌──────────────────────────────▼──────────────────────────────────────────┐
│  TIER 3 — Client Theme  (theme-{client}.css in consuming project)       │
│                                                                         │
│  Client brand overrides — same token names as Tier 2, different values. │
│  One file per client. Cascade wins. Components are untouched.           │
│                                                                         │
│  :root { --background-brand-primary: #2563EB; --font-family-heading: … }│
└─────────────────────────────────────────────────────────────────────────┘
```

**The key rule:** token names are fixed at Tier 2. Only values change between tiers. A component written against `--background-brand-primary` works for Brik's poppy red, any built-in theme, or any client brand — with zero code changes.

## Token types: primitives vs semantic

| Type | Purpose | Example | Used in |
|------|---------|---------|---------|
| **Primitive** | Raw scale value — the number or color itself | `--color-poppy-light: #e35335` | Never in components |
| **Semantic** | Purpose-bound alias pointing to a primitive | `--background-brand-primary: var(--color-poppy-light)` | Components, always |

Primitives define the palette. Semantics define the intent. Components only know intent.

```css
/* ✅ Correct — semantic, works across all themes */
.bds-button { background: var(--background-brand-primary); }

/* ❌ Wrong — hardcoded, breaks in every non-default theme */
.bds-button { background: #e35335; }

/* ❌ Wrong — primitive, breaks the abstraction */
.bds-button { background: var(--color-poppy-light); }
```

## Naming convention

```
--[category]-[role]-[variant]
```

| Category | Pattern | Examples |
|----------|---------|---------|
| Background | `--background-{role}` | `--background-brand-primary`, `--background-primary-hover`, `--background-info` |
| Text | `--text-{role}` | `--text-primary`, `--text-brand-primary`, `--text-info` |
| Surface | `--surface-{role}` | `--surface-primary`, `--surface-secondary`, `--surface-info` |
| Border | `--border-{role}` | `--border-primary`, `--border-brand-primary`, `--border-info` |
| Page | `--page-{role}` | `--page-primary`, `--page-secondary` |
| Font family | `--font-family-{role}` | `--font-family-heading`, `--font-family-body`, `--font-family-label` |
| Typography | `--{style}-{size}` | `--heading-lg`, `--body-md`, `--label-sm` |
| Spacing | `--padding-{size}`, `--gap-{size}` | `--padding-lg`, `--gap-md` |
| Border radius | `--border-radius-{size}` | `--border-radius-sm`, `--border-radius-md` |
| Shadow | `--shadow-{size}` | `--shadow-sm`, `--shadow-lg` |
| Scale step | `--{category}-{index}` | `--space-400`, `--font-size-700`, `--color-grayscale-dark` |

## Figma variable collections → CSS output

Figma organises variables into named collections. Each collection has modes. Style Dictionary flattens the active mode of each collection into `figma-tokens.css`.

| Figma collection | Active modes used | What it generates |
|-----------------|-------------------|-------------------|
| `primitives` | `value` | Raw scale: `--color-poppy-*`, `--space-*`, `--font-size-*`, `--border-radius-*`, `--shadow-blur-*` |
| `color` | `light` / `dark` | Semantic roles: `--background-*`, `--text-*`, `--surface-*`, `--border-*`, `--page-*` |
| `typography` | `default` | Composite tokens: `--heading-lg`, `--body-md`, `--label-sm`, `--font-family-*` |
| `spacing` | `default` | `--padding-*`, `--gap-*` |
| `border-radius` | `soft` | `--border-radius-sm/md/lg` |
| `border-width` | `default` | `--border-width-sm/md/lg` |
| `elevation` | `subtle` | `--shadow-sm/md/lg/xl`, `--blur-radius-*` |
| `breakpoint` | `default` | `--breakpoint-web/tablet/mobile` |

**Light vs dark mode:** `figma-tokens.css` uses the `light` mode of the `color` collection. `figma-tokens-dark.css` uses the `dark` mode. Both files use identical variable names — consuming projects import dark after light, and the browser applies whichever matches `[data-theme="dark"]`.

## Style Dictionary pipeline

Figma is the only place tokens are edited. The pipeline is:

```
1. Designer edits variables in Figma (Foundations file)
         ↓
2. Agent runs pull-variables.js with channel ID
   → /tmp/figma-vars.json  (raw Figma variable data)
         ↓
3. node scripts/sync-figma-mcp.js /tmp/figma-vars.json --build
   → patches design-tokens/tokens-studio.json (W3C DTCG format)
   → runs npm run build:all-tokens
         ↓
4. scripts/flatten-tokens-studio.js
   → design-tokens/tokens-figma.json  (flattened, mode-resolved)
         ↓
5. Style Dictionary
   → tokens/figma-tokens.css          ← imported by consuming projects
   → tokens/figma-tokens-dark.css     ← dark mode variant
   → build/figma/js/tokens.mjs        ← JS / TypeScript
   → build/figma/swift/*.swift        ← iOS
```

**Rule:** never manually edit `tokens/figma-tokens.css`. It is generated on every sync. Manual edits will be overwritten. Use `tokens/overrides.css` for gap-fill tokens that aren't in Figma yet.

## Consuming BDS in a project

Every Brik product and client project consumes BDS as a git submodule.

### 1. CSS import cascade (globals.css — order matters)

```css
/* 1. Web fonts */
@import '../brik-bds/tokens/fonts.css';

/* 2. Generated token values — light mode */
@import '../brik-bds/tokens/figma-tokens.css';

/* 3. Generated token values — dark mode (scoped to [data-theme="dark"]) */
@import '../brik-bds/tokens/figma-tokens-dark.css';

/* 4. Gap-fill and theme palette overrides */
@import '../brik-bds/tokens/overrides.css';

/* 5. Client brand — same token names, different values */
@import './styles/theme-{client}.css';
```

### 2. TypeScript token layer (src/lib/tokens.ts)

Never write raw `var(--...)` strings inline. Map every token to a typed export:

```ts
import { color, font, space, gap, size, border, shadow } from '@/lib/tokens';

// ✅ Correct
style={{ color: color.text.primary, fontSize: font.size.body.md }}

// ❌ Wrong — bypasses the type layer, breaks refactor
style={{ color: 'var(--text-primary)', fontSize: '16px' }}
```

The reference implementation lives in `brik-client-portal/src/lib/tokens.ts`. Copy it to every new project; update the entries to match what the client theme exposes.

### 3. Pre-commit enforcement

A token compliance gate blocks hardcoded hex colors, raw `var()` strings, and hardcoded `fontSize`/`rgba`/`padding` values at commit time. Copy `.husky/pre-commit` from `brik-client-portal` into every new project.

## Client theming

A client brand is a single CSS file (~12–27 token overrides). No components change. See [Client Theming](?path=/docs/foundations-design-tokens-client-theming--docs) for the full cascade, override tiers, known gaps, and font-only override patterns.

**Validate with Client Sim theme:** Switch to `★ Client Sim` in the Storybook toolbar. It assigns Georgia/Verdana/Courier New to heading/body/label families — any component using the wrong `--font-family-*` token becomes immediately visible.

---

## Built-in themes

BDS ships 9 themes. Switch using the paintbrush icon in the toolbar above. Each theme overrides semantic color tokens only — spacing, shadows, and component structure are theme-invariant. See [Client Theming](?path=/docs/foundations-design-tokens-client-theming--docs) for the full theme gallery and how to add a custom brand.

## Adding a new token

**Always start in Figma — never in CSS.**

1. Add the variable to the correct collection in the Figma Foundations file
2. Connect Claude with the Figma dev plugin (provide channel ID)
3. Run the sync: `bun pull-variables.js <channel> > /tmp/vars.json && node scripts/sync-figma-mcp.js /tmp/vars.json --build`
4. Verify the token appears in `tokens/figma-tokens.css`
5. Add it to the consuming project's `src/lib/tokens.ts` export map

If the token is a semantic alias (references another variable), it must be added manually to `design-tokens/tokens-studio.json` under `color/light` and `color/dark` after the sync — the pipeline only patches primitive values automatically.

**Exception:** structural gap-fill tokens that don't belong in Figma (e.g., theme-specific palette values) go in `tokens/overrides.css`.

## What NOT to do

```css
/* ❌ Never use a primitive in a component */
.card { color: var(--color-grayscale-darkest); }
/* ✅ Use the semantic alias */
.card { color: var(--text-primary); }

/* ❌ Never hardcode a value */
.button { background: #e35335; }
/* ✅ Use the semantic token */
.button { background: var(--background-brand-primary); }

/* ❌ Never edit figma-tokens.css manually */
/* ✅ Edit in Figma → run sync → regenerates automatically */

/* ❌ Never skip the TypeScript layer in consuming projects */
style={{ color: 'var(--text-primary)' }}
/* ✅ Import from the token map */
import { color } from '@/lib/tokens';
style={{ color: color.text.primary }}
```
